第一章 系统概述

### 1.1 技术定位

心屿AI情感助手是心理健康辅助平台"心屿"的核心功能模块，基于

大语言模型（Large Language Model, LLM）构建，旨在为大学生用户

提供7x24小时不间断的情绪倾听与心理支持服务。

系统采用"预训练大模型 + Prompt Engineering + In-Context Learning"

的技术架构，区别于传统的微调（Fine-tuning）路线，在保证专业性的

前提下显著降低了开发与部署成本。

### 1.2 技术选型依据

经过对主流大语言模型的横向对比评估，本系统选用小米MiMo v2.5

作为基础推理模型，选型依据如下：

评估维度        MiMo v2.5    GPT-4o      文心一言     通义千问

─────────────────────────────────────────────────────────────

中文对话能力     优           优          良          良

角色扮演一致性   优           优          中          良

API调用延迟      低(国内)     高(海外)    低(国内)    低(国内)

千Token成本      低           高          中          中

数据合规性       符合         存疑        符合        符合

指令遵循能力     优           优          良          良

综合考量中文对话质量、响应延迟、数据合规及成本因素，MiMo v2.5

为当前阶段的最优选择。

第三章 知识库架构：分层知识注入体系

本系统不采用传统的向量数据库检索增强生成（Retrieval-Augmented

Generation, RAG）架构，而是构建了一套"分层知识注入体系"，将

专业知识通过三个层级注入模型推理过程。

### 3.1 知识层级架构

Layer 1: System Prompt层（静态知识）

├─ 角色定义与身份锚定

├─ 行为规范与能力边界

├─ 安全护栏与危机干预规则

└─ 输出风格与格式约束

Layer 2: In-Context Learning层（动态知识）

├─ 最近10轮对话历史（约2000 tokens）

├─ 用户情绪状态隐式建模

└─ 话题连贯性维持

Layer 3: 领域知识层（规则引擎）

├─ 心理测评量表（PHQ-9/GAD-7/PSS/睡眠）

├─ 评分算法与风险分级

├─ 结构化报告生成模板

└─ 紧急联系人与热线数据库

### 3.2 Layer 1: System Prompt设计

System Prompt是AI行为的核心控制器，完整内容如下：

┌─────────────────────────────────────────────────────────┐

│ "你是一个温暖贴心的AI情感助手，名叫'心语'。            │

│  你的任务是倾听用户的心声，给予温暖的回应和实用的建议。 │

│  你会：                                                  │

│  1. 认真倾听，表示理解和共情                             │

│  2. 给出具体可行的缓解情绪的小方法                       │

│  3. 语气温柔亲切，像朋友一样                             │

│  4. 如果用户情绪很严重，温柔地建议寻求专业帮助           │

│  5. 回复简洁温暖，不要太长"                              │

└─────────────────────────────────────────────────────────┘

Prompt设计遵循以下工程原则：

(1) 角色锚定（Role Anchoring）

首句明确定义AI身份："你是..."，利用LLM的指令遵循能力，

在整个对话过程中维持角色一致性。

(2) 行为约束（Behavioral Constraint）

通过"你会"列表（共5条规则）划定AI行为边界，防止角色漂移

（Role Drift）。

(3) 安全护栏（Safety Guardrail）

第4条规则内置危机干预逻辑，当AI感知到用户情绪严重时，

自动触发专业转介机制。

(4) 风格控制（Style Control）

第5条规则限定输出长度，避免冗余回复影响用户体验。

(5) 去医疗化声明（De-medicalization）

Prompt中刻意避免"诊断"、"治疗"、"处方"等医疗术语，

将AI定位为"情感陪伴者"而非"医疗提供者"，规避法律风险。

### 3.3 Layer 2: In-Context Learning机制

In-Context Learning（ICL）是本系统的核心AI技术策略。

技术原理：

大语言模型具备"少样本学习"（Few-shot Learning）能力，即在

不修改模型参数的前提下，通过在输入上下文中提供示例，引导模型

生成符合预期的输出。

本系统的ICL实现：

(1) 对话历史检索

每次用户发送消息，系统从chat_history表检索最近10轮对话

（按时间倒序，LIMIT 10），反转为正序后作为上下文注入。

(2) 上下文窗口管理

10轮对话 ≈ 20条消息 ≈ 2000 tokens

占MiMo v2.5输出预算（1024 tokens）的约50%，留有足够空间

生成高质量回复。

(3) 动态上下文策略

- 用户首次对话：仅System Prompt + 当前消息

- 多轮对话：System Prompt + 历史消息 + 当前消息

- 长对话截断：保留最近10轮，丢弃早期对话

(4) 角色标注一致性

历史消息中的role字段严格映射：

chat_history.role='user' → messages.role='user'

chat_history.role='assistant' → messages.role='assistant'

### 3.4 Layer 3: 领域知识注入

心理测评模块采用规则引擎实现，不依赖AI推理：

(1) 量表知识库

量表名称    来源                    题数  评分级别  总分范围

──────────────────────────────────────────────────────

PHQ-9       APA抑郁症筛查工具       9     4级(0-3)  0-27

GAD-7       APA焦虑症筛查工具       7     4级(0-3)  0-21

PSS         Cohen压力感知量表       10    5级(0-4)  0-40

睡眠评估    PSQI简化版              5     4级(0-3)  0-15

(2) 评分算法

// PHQ-9 风险分级

score ∈ [0, 4]   → 无抑郁（Normal）

score ∈ [5, 9]   → 轻度抑郁（Mild）

score ∈ [10, 14] → 中度抑郁（Moderate）

score ∈ [15, 19] → 中重度抑郁（Moderately Severe）

score ∈ [20, 27] → 重度抑郁（Severe）

// GAD-7 风险分级

score ∈ [0, 4]   → 无焦虑（Normal）

score ∈ [5, 9]   → 轻度焦虑（Mild）

score ∈ [10, 14] → 中度焦虑（Moderate）

score ∈ [15, 21] → 重度焦虑（Severe）

// PSS 风险分级（含反向计分题）

score ∈ [0, 13]  → 低压力（Low）

score ∈ [14, 26] → 中等压力（Moderate）

score ∈ [27, 40] → 高压力（High）

(3) 报告生成模板

每个量表 × 每个风险等级 = 一套结构化报告模板

报告包含：得分解读 + 风险等级 + 专业建议 + 转介指引

总计约15套报告模板，通过规则引擎动态填充生成

第四章 AI推理流程详解

### 4.1 完整推理链路

Step 1: 用户输入接收

前端 → POST /api/chat {message, gender}

Step 2: 消息持久化

INSERT INTO chat_history (role, message, gender)

VALUES ('user', <用户消息>, <性别>)

Step 3: 上下文检索

SELECT role, message FROM chat_history

WHERE gender = <性别>

ORDER BY created_at DESC LIMIT 10

→ 反转为正序

Step 4: Prompt构建

system = System Prompt（角色定义 + 行为规范）

messages = [历史消息1, 历史消息2, ..., 当前消息]

Step 5: API调用

POST https://token-plan-cn.xiaomimimo.com/anthropic/v1/messages

Headers: x-api-key, Content-Type, anthropic-version

Body: {model, system, max_tokens:1024, messages}

Step 6: 响应解析

解析content数组，提取type='text'的文本内容

异常处理：API失败时返回预设安慰性回复

Step 7: 回复持久化

INSERT INTO chat_history (role, message, gender)

VALUES ('assistant', <AI回复>, <性别>)

Step 8: 前端渲染

返回JSON {reply: <AI回复>}

前端渲染为聊天气泡

### 4.2 异常处理机制

异常类型              处理策略

──────────────────────────────────────────────

API超时(>30s)         返回预设回复 + 热线信息

API返回错误           返回预设回复 + 热线信息

响应内容为空           返回"抱歉，我现在有点累了"

JSON解析失败          返回预设回复

数据库写入失败         静默失败，不影响用户体验

### 4.3 安全检测机制

(1) 关键词风险检测

高风险关键词库：

["自杀", "想死", "不想活", "结束生命", "自伤",

"割腕", "伤害自己", "活不下去", "没有意义", "累死了"]

检测逻辑：用户消息包含任意关键词 → AI回复附加危机干预信息

(2) Prompt安全规则

System Prompt第4条规则实现AI行为层面的安全防护：

"如果用户情绪很严重，温柔地建议寻求专业帮助"

触发条件：AI通过语义理解感知用户情绪严重程度

响应方式：温和建议 + 推荐心理援助热线（12320-5）

### 7.2 数据安全措施

- API Key不暴露于前端，仅在后端服务器存储

- 用户对话数据仅存储于自有服务器，不上传至第三方

- 管理后台访问需bcrypt密码验证

- Flask-Limiter限流防止API滥用

- 会话Token单设备绑定，防止账号共享

第八章 性能优化与成本控制

### 8.1 API调用优化

- 对话历史限制为10轮，控制单次请求token量

- max_tokens=1024，限制AI输出长度

- 30秒超时，避免长时间等待

- 限流10次/分钟，防止滥用

### 8.2 缓存策略

- 心情记录：localStorage本地缓存

- 测评历史：localStorage本地缓存

- 番茄钟数据：localStorage本地缓存

- 仅在需要同步时调用服务器API

### 8.3 成本估算

单次AI对话成本：

输入tokens ≈ 2000（System Prompt + 历史消息）

输出tokens ≈ 200（平均回复长度）

MiMo v2.5定价 ≈ ¥0.002/千tokens

单次成本 ≈ ¥0.004

月度成本估算（假设1000日活用户，每人5轮对话）：

月对话次数 = 1000 × 5 × 30 = 150,000次

月API成本 ≈ 150,000 × ¥0.004 = ¥600

第九章 技术创新点总结

创新点1：分层知识注入架构

区别于传统RAG方案，构建System Prompt + ICL + 规则引擎的

三层知识体系，兼顾专业性与灵活性。

创新点2：Prompt Engineering替代微调

通过精心设计的System Prompt实现角色锚定，无需准备标注数据集

和GPU训练资源，将模型迭代周期从天级缩短至秒级。

创新点3：In-Context Learning情绪感知

利用10轮对话历史实现动态上下文，AI能根据用户情绪变化

实时调整回复策略，实现"伪个性化"效果。

创新点4：规则引擎与AI协同

心理测评采用规则引擎保证评分客观性，AI对话采用大模型保证

回复灵活性，两者协同构建完整的心理健康服务链路。

创新点5：安全护栏设计

在System Prompt中嵌入危机干预规则，结合关键词检测，

构建双重安全防护机制。

A.2 作品概述

心屿是一款面向大学生的心理健康辅助Web应用平台。系统名称“心屿”取自“心灵港湾”之意，寓意为用户提供一个安全、温暖的心灵栖息之所。

平台集成了AI情感对话、心情记录与追踪、标准化心理量表测评、匿名社区分享、情感互助匹配、学习辅助工具等核心功能模块。通过温暖的视觉设计和专业的心理健康服务，帮助大学生更好地了解自己、管理情绪、获得支持。

A.3 核心功能模块

A.3.1 AI情感助手

基于小米MiMo v2.5大语言模型的AI情感对话助手，通过System Prompt角色设定技术，AI扮演名为“心语”的情感助手角色。系统维护最近10轮对话历史作为上下文，实现连贯的多轮对话。内置危机检测机制，当识别到用户高风险表达时，自动引导用户寻求专业心理援助。

A.3.2 情绪中心

提供可视化心情转盘交互，支持8种基本情绪（开心、难过、生气、焦虑、平静、兴奋、疲惫、一般）的记录。结合Canvas绘制的情绪趋势图，帮助用户追踪情绪变化。创新性地引入基于心情的美食推荐功能，根据情绪状态智能匹配6类美食。

A.3.3 心理测评

集成4个国际标准心理量表：PHQ-9抑郁量表（9题）、GAD-7焦虑量表（7题）、PSS压力量表（10题）、睡眠质量评估（5题）。前端JavaScript实现评分算法，根据得分自动计算风险等级，生成结构化评估报告和专业建议。支持综合心理报告生成。

A.3.4 社区广场

匿名社区分享平台，包含树洞倾诉和情感互助两大板块。树洞板块支持匿名发帖、点赞、评论，提供最新/最热两种排序。情感互助板块实现基于标签相似度的匿名匹配系统，匹配成功后可进行一对一匿名聊天。

A.3.5 学习广场

集成三大学习辅助工具：笔记管理（新建/编辑/删除/导出）、番茄钟（25分钟专注/5分钟短休/15分钟长休）、课表管理（按周分标签展示，支持PDF/Word/Excel导入）。

A.3.6 个人中心

包含深色模式切换、紧急联系人设置、心理求助热线、安全问题密码找回、治愈信箱（给未来自己写信）、数据报告、每日打卡等功能。内置虚拟桌宠系统，宠物等级与用户互动关联。

A.3.7 管理后台

独立管理后台，包含数据概览、用户管理（封禁/解封/删除）、社区帖子管理、AI对话记录管理、心情统计、事件日志、情感互助管理、学生信息管理等8大功能模块。桌面端侧边栏导航，移动端顶部Tab切换。

A.4 技术架构

A.4.1 前端技术栈

HTML5 / CSS3 / JavaScript（ES6+），无框架依赖

单页应用（SPA）架构，页面切换无需刷新

CSS变量主题系统，支持深色/浅色模式一键切换

响应式设计，最大宽度480px，适配手机和桌面端

Canvas 2D API绘制情绪趋势折线图

CSS conic-gradient实现心情转盘交互

localStorage本地缓存，减少服务器请求

A.4.2 后端技术栈

Python 3.12 + Flask Web框架

Gunicorn WSGI服务器（多worker进程，端口5000）

SQLite3数据库（轻量级，零配置）

bcrypt密码哈希（注册/登录/管理员）

Flask-Session服务端会话管理

Flask-Limiter API限流保护（10次/分钟）

RESTful API设计，共计约45个接口

A.4.3 AI技术栈

小米MiMo v2.5大语言模型（云端API调用）

System Prompt角色设定技术（情感助手“心语”）

In-Context Learning上下文学习（最近10轮对话历史）

关键词风险检测 + Prompt安全护栏双重防护

A.4.4 部署架构

操作系统：Ubuntu Linux 22.04 LTS

Web服务器：Nginx反向代理（静态文件 + API转发）

进程管理：Systemd服务管理（自动重启）

A.5 数据库设计

共14个核心数据表：

### 1. server_users — 注册用户表（用户名、密码、昵称、性别、头像、封禁状态、会话令牌、安全问题）

### 2. moods — 心情记录表（心情类型、备注、性别）

### 3. checkins — 打卡记录表（日期、完成状态）

### 4. chat_history — AI对话历史表（角色、消息内容、性别）

### 5. anonymous_posts — 匿名帖子表（内容、匿名昵称、设备ID、媒体URL）

### 6. anonymous_replies — 帖子评论表（帖子ID、内容、匿名昵称）

### 7. anonymous_likes — 点赞记录表（帖子ID、设备ID）

### 8. user_devices — 设备用户表（设备ID、昵称、首次/最近访问时间）

### 9. user_events — 用户事件表（设备ID、事件类型、事件数据）

10. test_results_sync — 测评结果同步表（用户名、测评ID、得分、等级）

11. support_profiles — 互助档案表（设备ID、匿名名、心情标签、兴趣标签）

12. support_matches — 互助匹配表（用户1、用户2、状态）

13. support_messages — 互助消息表（匹配ID、发送者、内容、已读状态）

14. students — 学生信息表（学号、姓名、性别、年龄、专业、班级）

心情管理（2个）：获取心情记录、提交心情记录

打卡系统（2个）：获取打卡记录、提交打卡

社区功能（8个）：帖子CRUD、点赞/取消点赞、评论CRUD

情感互助（10个）：档案管理、发现用户、匹配、聊天、搜索、解除匹配、举报

学习工具（1个）：课表导入

测评系统（2个）：同步测评结果、获取测评历史

A.7 技术创新点

### 1. 分层知识注入架构：System Prompt + In-Context Learning + 规则引擎三层知识体系，兼顾专业性与灵活性

### 2. Prompt Engineering替代微调：通过System Prompt实现角色锚定，无需标注数据和GPU训练，迭代周期从天级缩短至秒级

### 3. In-Context Learning情绪感知：10轮对话历史实现动态上下文，AI根据用户情绪变化实时调整回复策略

### 4. 规则引擎与AI协同：心理测评采用规则引擎保证客观性，AI对话采用大模型保证灵活性，协同构建完整服务链路

### 5. 安全护栏设计：System Prompt危机干预规则 + 关键词检测双重防护，确保用户安全

### 6. 心情转盘交互创新：CSS conic-gradient + 角度映射算法，实现直觉化情绪记录交互

### 7. 情感互助匹配系统：基于标签相似度的匿名匹配算法，帮助用户找到有相似经历的伙伴

### 8. 响应式单页应用：纯HTML/CSS/JS实现SPA架构，无框架依赖，轻量高效
